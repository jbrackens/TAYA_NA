================================================================================
KEY DESIGN DECISIONS
================================================================================

1. UUID PRIMARY KEYS
   - Reason: Database-agnostic, distributed system friendly
   - Generated using gen_random_uuid()
   - All 18 tables use UUID PK

2. NUMERIC(18,2) FOR FINANCIAL
   - Reason: Exact decimal arithmetic, prevents float errors
   - Supports up to 999,999,999,999,999.99
   - Used in: balance, stake, odds, amount fields

3. ENUM TYPES
   - Reason: Database-level constraint enforcement
   - 16 enums total for status and type fields
   - Enables query optimization and prevents invalid values

4. JSONB FOR EXTENSIBILITY
   - Reason: Future-proof schema without migrations
   - Used for: metadata, criteria, targeting_rules, reward_config
   - Provides: Query flexibility + validation layer in code

5. AUDIT LOG PARTITIONING
   - Reason: Monthly partitions enable efficient archival
   - Automated partition creation via trigger function
   - Facilitates retention policies (e.g., keep 2 years)

6. CASCADING DELETES
   - Reason: Maintain data consistency and audit trail
   - CASCADE: Removes dependent records (transactions, bets, etc.)
   - RESTRICT: Prevents deletion if critical dependencies exist
   - SET NULL: Soft references for historical tracking

7. COMPOSITE INDEXES
   - Reason: Optimize multi-column filter queries
   - Examples: (user_id, status), (leaderboard_id, score DESC)
   - Reduces query execution time for common patterns
