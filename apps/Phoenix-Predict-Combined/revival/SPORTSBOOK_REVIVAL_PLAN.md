## 1. Mission
Revive the Phoenix platform as a production-grade sportsbook system with modern engineering standards, delivering both:
1. Customer-facing Sportsbook frontend
2. Talon Backoffice frontend

while replacing the legacy Scala/Akka backend with a modern Go backend and preserving behavioral parity for core betting and wallet workflows.

---

## 2. Locked Strategic Decisions
1. **Backend direction:** Migrate off Akka to **Go**.
2. **Runtime targets:** **Node 20+** for frontends/tooling and **Java 21** for transitional legacy runtime only.
3. **First launch scope:** **Sportsbook + Talon backoffice together**.
4. **Migration strategy:** Strangler pattern with dual-run and parity verification before cutover.
5. **Data scope for relaunch:** No legacy dataset migration/onboarding; launch on clean-state schema + deterministic seed data + live write reconciliation.

---

## 3. North Star

### Product North Star
1. A user can register/login, view markets, place bets, and see outcomes end-to-end.
2. Operations can manage users/markets through Talon backoffice with full auditability.
3. Platform is releasable and operable without legacy Flipadmin dependencies.

### Engineering North Star
1. One-command local bring-up for full stack.
2. Reproducible CI pipeline with deterministic builds and tests.
3. Security baseline: no critical vulns, no hardcoded secrets, SBOM generated.
4. Observability baseline: logs, metrics, tracing, health checks, alertable signals.
5. Safe deployability: rollback tested with rehearsed cutover/recovery paths.

### Launch Readiness KPIs
1. Critical E2E flow pass rate: **100%** in staging.
2. P1 incident count in dress rehearsal: **0**.
3. Deployment rollback test: **pass**.
4. API parity on critical endpoints: **>= 99.5%** expected behavior match.
5. Ledger reconciliation variance: **0 unresolved critical mismatches**.

---

## 4. Scope and Non-Goals

### In Scope
1. Phoenix backend replacement to Go.
2. Sportsbook + Talon integration with new backend.
3. Live-write data integrity hardening and operational readiness (no legacy data migration in relaunch scope).
4. CI/CD, security, and observability modernization.

### Out of Scope (Initial Relaunch)
1. Full feature expansion beyond legacy parity.
2. Rust backend implementation (revisit post-launch only for hotspots).
3. New market/geography launch features not needed for core relaunch.

## 7. Delivery Model
1. 2-week sprints.
2. Codex executes implementation tasks directly.
3. Sprint closes only with evidence: changed files, test output summary, risk updates.
4. Every sprint updates this file in three places:
   - `North Star Status`
   - `Backlog`
   - `Progress Log`

### Documentation Cadence (Always-On)
1. Every material code or script change must be accompanied by a doc/report update in `revival/` in the same work session.
2. Every verification command run for gating (`make verify*`, `make security*`, `make qa*`, `make release-*`) must have a traceable artifact path captured in markdown.
3. At end of each execution block, update:
   - backlog item statuses affected
   - `North Star Status` snapshot deltas
   - `Progress Log` journal entry with blockers and next step
4. If a gate is flaky or environment-dependent, document:
   - exact failure mode
   - mitigation/fallback in scripts
   - residual risk carried to next phase
